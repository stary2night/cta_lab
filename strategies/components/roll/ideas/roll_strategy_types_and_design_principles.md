# Roll Strategy 类型归纳与 Roll Component 设计原则

这份文档先不讨论具体代码接口，只从业务抽象层面回答两个问题：

1. `Roll Strategy` 大致有哪些典型类型
2. 设计 `Roll Component Layer` 时，应优先考虑哪些基本原则

目标是先把大方向讲清楚，作为 `Phase P2` 进一步细化设计的基础。

---

## 一、我们目前理解到的 Roll Strategy 类型

基于 GMAT3 复现、已有 CTA 研究脚本，以及最近补充的 roll strategy 资料，
目前可以把 `Roll Strategy` 粗略归纳为下面几类。

### 1. 主力跟踪型

这是最常见的一类。

核心特点：
- 先定义“主力合约”如何判定
- 再定义何时从旧主力切换到新主力
- 再定义展期期间如何分日移仓

典型规则：
- OI 最大
- 成交量最大
- 固定交割月映射
- 最后持有日前 N 天触发换仓

典型应用：
- GMAT3 中的大部分单资产子组合
- 常规国内商品、股指、国债的主力连续策略

这一类策略的关键不只是“选主力”，还包括：
- 主力锁定期
- 展期执行方式
- 展期收益定义

### 2. 动态选月型

这一类策略不满足于“近月 -> 次主力”的简单切换，
而是在可选合约集合里主动挑选更优月份。

核心特点：
- 候选集合不止 old/new 两个合约
- 会根据期限结构、隐含 roll yield、流动性等因素选目标合约
- 有时会加入“继续持有原合约”的低换手约束

典型应用：
- `S&P GSCI Dynamic Roll`
- 各类 curve-aware / carry-aware 的商品展期策略

这一类策略说明：
- `Roll Strategy` 不能只抽象成“主力合约序列”
- 它更像“目标合约规划策略”

### 3. 执行优化型

这一类策略的核心不在于换哪个合约，而在于“怎么换”。

核心特点：
- old/new 合约可能已经确定
- 真正的 alpha 或效率来自展期执行方式
- 会关注：
  - 执行成本
  - 流动性
  - 执行风险
  - 分天交易安排

典型应用：
- MVE roll
- Goldman-style multi-day roll
- 其他分布式执行策略

这一类策略说明：
- `Roll Strategy` 必须把“执行层”当作独立维度
- 不能把所有展期逻辑都塞进选约规则里

### 4. 替代标的切换型

这一类策略不仅处理期货合约，还处理“期货上市前/不可交易阶段”的替代资产。

核心特点：
- 前半段可能使用现货指数、债券指数、代理指数
- 后半段切换到期货展期收益
- 切换逻辑本身就是策略定义的一部分

典型应用：
- GMAT3 中部分境内股指、国债子组合

这一类策略说明：
- `Roll Strategy` 的输出不只是期货净值链
- 而是更一般的资产级价值序列

### 5. 复合资产型

这一类策略的输出资产，并不是单一品种，而是由多个底层品种共同构成。

核心特点：
- 底层可能有多个 component
- component 自己各有 roll 逻辑
- 上层还要再做 component 权重整合

典型应用：
- GMAT3 的 `BLACK`
- 未来可能的 sector/basket roll strategy

这一类策略说明：
- `Roll Strategy` 不一定等于“单品种策略”
- 平台要能支持“复合底层资产”的 value series 构造

## 二、一个重要判断

综合上面几类，我们现在更清楚地看到：

**Roll Strategy 不是单一规则，而是一个由多个决策层拼起来的资产生成过程。**

至少可能包含：
- 候选合约集合定义
- 目标合约选择
- 展期触发判断
- 展期执行计划
- 收益率生成
- 资产级价值序列构造
- 可穿透的底层持仓映射

因此，未来平台里的 `Roll Component Layer` 不应被理解成“连续合约扩展函数”，
而应被理解成：

**一个面向资产级策略生成的组件层。**

## 三、设计 Roll Component 时的基本原则

下面这些原则，建议作为后续 P2 设计时的硬约束或优先考虑项。

### 原则 1：先抽象流程，再抽象规则

不要一开始按资产类型拆一堆类。

更重要的是先把 Roll Strategy 的共性流程拆出来：
- 候选集合
- 合约选择
- 展期执行
- 收益生成
- 价值序列构造

然后再思考每一步有哪些配置差异、哪些真的需要插件。

### 原则 2：优先“基类 + profile + 可插拔组件”

当前更适合的方向不是：
- `DomesticEquityRollStrategy`
- `DomesticBondRollStrategy`
- `OverseasFuturesRollStrategy`

而是：
- 一个 `RollStrategyBase`
- 多个 `RollStrategyProfile`
- 少量组件型插件

也就是说：
- 差异优先通过 profile/config 表达
- 真正逻辑差异很大时再用独立组件

### 原则 3：输出对象必须足够完整

未来 Roll Strategy 不应该只输出一条净值曲线。

至少要考虑输出：
- `contract_plan`
- `roll_schedule`
- `roll_return`
- `value_series`
- `lookthrough_book`

如果只输出最终 `value_series`，后续就很难支持解释、调试和交易落地。

### 原则 4：必须支持可穿透性

未来任何上层配置策略都会把 Roll Strategy 生成的资产当作“可投资对象”。

但真正交易时，必须能继续向下看到：
- 当前底层到底持有哪些合约
- 每个合约比例是多少
- 是否处于展期期间
- old/new 合约的权重如何分配

因此可穿透性不是附加功能，而是设计前提。

### 原则 5：要区分“选什么”和“怎么换”

这是从论文和动态选月策略里得到的一个关键启发。

未来设计里必须把这两件事分开：
- 目标合约选择
- 展期执行方式

否则很多高级 roll strategy 会被混在同一个逻辑块里，很难扩展。

### 原则 6：要把“资产级价值序列”当作一等产物

Roll Strategy Layer 的真正价值，不只是生成中间表，
而是为上层 `Cross Asset Allocation` 提供“可投资的资产”。

因此，最终产物应优先围绕：
- 可投资资产
- 资产价值序列
- 资产穿透构成

来设计，而不是只围绕单日合约切换逻辑。

### 原则 7：暂时不要过度设计更远的引擎问题

例如：
- 是否独立成 `IndexEngine`
- 是否单独抽象 `Multi-Currency Engine`

这些问题现在还不够清晰。

目前更合理的做法是：
- 先把 Roll Strategy Layer 自己设计清楚
- 等上层 Portfolio / Look-Through 抽象更稳定之后，再决定这些更远层的问题

## 四、当前阶段对 P2 的最直接启发

如果把上面的判断压缩成一句最直接的话，就是：

**P2 的目标不是“做一个更强的 continuous”，而是“做一个能生成资产级 roll strategy 的组件层”。**

这个组件层未来至少要能支持：
- 主力跟踪型
- 动态选月型
- 执行优化型
- 替代标的切换型
- 复合资产型

而且它必须天然支持：
- profile 化
- 组件化
- 可穿透性

---

## 五、下一步最适合继续细化的内容

在这份高层归纳之后，下一步最适合细化的是：

1. `RollStrategyBase` 应该负什么责任
2. `RollStrategyProfile` 至少要有哪些字段
3. `Selector / Executor / Composer` 的边界怎么拆
4. `RollStrategyResult` 至少应包含哪些输出对象

这些内容更适合进入 `drafts/` 目录继续展开。
